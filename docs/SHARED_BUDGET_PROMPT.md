# SharedBudget — детальный промпт для реализации MVP

Используй этот документ как единый источник требований: для разработки с нуля, для code review и для промптов к ИИ-ассистентам. Цель — **MVP за ~2 недели** реальной разработки.

---

## 1. Продукт и ценность

**SharedBudget** — веб-приложение для учёта общих трат и взаимных долгов в небольших группах (квартира, поездка, проект).

**Пользовательский сценарий:**

1. Пользователь регистрируется и входит в систему.
2. Создаёт **пространство** (например: «Квартира на Ленина», «Тур в Грузию»).
3. Приглашает участников **по ссылке-приглашению** (join link).
4. Участники вносят **траты**: кто заплатил, сумма, как делить между участниками.
5. Система показывает **балансы** (кто кому должен) и **минимальный набор переводов** для взаиморасчёта (settle-up).
6. Доступна **история операций** с **фильтрами по периоду**.
7. Управление пространством — через **настройки пространства** (отдельная админ-панель не нужна).

## 2. Объём MVP (обязательно)

### 2.1 Аутентификация

- Регистрация и вход (email + пароль или согласованный минимум — зафиксировать в реализации).
- **JWT** для доступа к API; **refresh-токен** — по желанию (если успеваете в срок MVP).

### 2.2 Пространства (Spaces)

- Создание пространства одним пользователем.
- **Роли:** **владелец** (owner) и **участник** (member).
- Владелец: переименование/удаление пространства (или мягкое архивирование — на усмотрение, но поведение должно быть предсказуемым), управление участниками в рамках настроек.
- Приглашение по **уникальной ссылке** (токен в URL); переход по ссылке — вступление в пространство (с учётом авторизации).

### 2.3 Траты (Expenses)

Поля и логика:

- **Сумма** (одна валюта на пространство в MVP — достаточно; мультивалюта — out of scope).
- **Кто заплатил** — один участник пространства (payer).
- **Как делить:**
  - **Поровну** между выбранными участниками (или всеми — уточнить в UI, но логика должна быть явной).
  - **По долям** (weights): каждому участнику доля; сумма долей должна сходиться с суммой траты (валидация на бэкенде).

Каждая трата должна порождать понятную запись в **истории** (кто создал/изменил — желательно для аудита; минимум — дата и состав полей).

### 2.4 Балансы и settle-up

- На основе трат посчитать **чистый баланс** каждого участника относительно группы (кто в плюсе / в минусе).
- Реализовать расчёт **минимального набора переводов** между участниками, чтобы все балансы сошлись в ноль (классическая задача **splitwise-style settle up** / минимизация числа транзакций при погашении долгов).
- Отобразить пользователю: «переведи X от A к B» (список рекомендуемых переводов).

### 2.5 История и фильтры

- Список **транзакций/операций** (траты как минимум; при необходимости — корректировки в будущем).
- **Фильтр по периоду** (дата от / дата до).

### 2.6 Настройки пространства

- Экран (или секция) настроек без отдельной «админки продукта»: название, участники, ссылка-приглашение (пересоздание токена — опционально), выход из пространства.

---

## 3. Нефункциональные требования MVP

- Данные изолированы по пространству: пользователь не видит чужие пространства без членства.
- API защищён авторизацией; проверка членства в пространстве на каждой операции с данными этого пространства.
- Валидация входных данных на бэкенде (схемы); на фронте — согласованные формы.

---

## 4. Технологический стек (одна строка для резюме)

**Frontend:** React (Vite) + TypeScript, TanStack Query (React Query), React Router, формы: React Hook Form + Zod.

**Backend:** Node.js + **NestJS** или **Express** (Express — если нужен более простой старт) + TypeScript.

**БД:** PostgreSQL + **Prisma** (ORM, миграции).

**Деплой:** локально — **Docker Compose** (app + Postgres); прод — **Railway** или **Render** (бэкенд + БД) + **Vercel** или **Netlify** (фронтенд).

---

## 5. Рекомендуемая структура репозитория

На усмотрение команды, типичный вариант:

- `apps/web` — Vite + React
- `apps/api` — NestJS или Express
- `packages/shared` (опционально) — общие типы/DTO между фронтом и бэком
- `docker-compose.yml` — Postgres + сервисы
- `docs/` — спецификация и архитектурные заметки

## 6. Модель данных (ориентир для Prisma)

Минимальный набор сущностей (имена полей — пример):

- **User** — id, email, passwordHash, createdAt
- **RefreshToken** (если делаете refresh) — userId, token, expiresAt
- **Space** — id, name, ownerId, inviteToken (уникальный), createdAt
- **SpaceMember** — spaceId, userId, role (`OWNER` | `MEMBER`), joinedAt
- **Expense** — id, spaceId, amount, payerId, description (опционально), createdAt, updatedAt
- **ExpenseSplit** — expenseId, userId, mode (`EQUAL` | `WEIGHT`), weight (для долей; для equal можно 1 или NULL по конвенции)

Индексы по `spaceId`, `userId`, датам для фильтров.

---

## 7. API (черновой контракт)

Реализовать REST (или tRPC — только если команда единогласна; в стеке указан классический подход).

Примеры эндпоинтов:

- `POST /auth/register`, `POST /auth/login`, (опционально) `POST /auth/refresh`, `POST /auth/logout`
- `GET /me`
- `POST /spaces`, `GET /spaces`, `GET /spaces/:id`, `PATCH /spaces/:id`, `DELETE /spaces/:id` (или archive)
- `POST /spaces/:id/join` (body: invite token или token из query при deep link)
- `GET /spaces/:id/members`
- `POST /spaces/:id/expenses`, `GET /spaces/:id/expenses?from=&to=`, `GET /spaces/:id/expenses/:expenseId`, `PATCH`, `DELETE`
- `GET /spaces/:id/balances`
- `GET /spaces/:id/settlements` (или вложено в balances) — список предлагаемых переводов

Ответы — JSON; ошибки — единый формат (код, сообщение).

---

## 8. Алгоритм settle-up (требование к реализации)

- Вход: массив балансов по userId (положительный = ему должны, отрицательный = он должен).
- Выход: список пар (from, to, amount) с минимальным числом переводов (жадные стратегии с двумя указателями на отсортированных должниках/кредиторах обычно достаточны).
- Покрыть unit-тестами на краевые случаи: все нули, один должник/один кредитор, много мелких долгов.

---

## 9. UI/UX MVP

- Простая навигация: список пространств → дашборд пространства (балансы + кнопка «добавить трату» + settle-up + история).
- Формы с Zod-схемами и отображением ошибок сервера.
- Состояние загрузки/ошибок через React Query.
- Без обязательной тёмной темы и пиксель-перфекта — но читаемо и последовательно.

---

## 10. Вне скоупа MVP (явно)

- Мобильные нативные приложения.
- Мультивалюта и курсы.
- Вложения/чеки.
- Уведомления email/push.
- Сложные права (кроме owner/member).
- Платежи через платёжные провайдеры (только «рекомендации к переводу»).

## 11. Критерии готовности MVP

- [ ] Регистрация/логин работает end-to-end с реальной БД.
- [ ] Создание пространства, приглашение, вступление вторым пользователем.
- [ ] CRUD трат (минимум создание + список + редактирование или удаление — на усмотрение, но не «только create»).
- [ ] Деление поровну и по долям с валидацией.
- [ ] Балансы и settle-up совпадают с ручной проверкой на тестовых данных.
- [ ] Фильтр истории по датам.
- [ ] Docker Compose поднимает Postgres и API (и при желании web).
- [ ] README с инструкцией запуска и переменными окружения.

## 12. Режим пошаговой работы (обязательно для ИИ)

При реализации проекта по этому документу ассистент **не ведёт всю разработку одним непрерывным потоком**. Действует пошаговый цикл:

1. **Декомпозиция** — заранее согласовать с пользователем (или взять из очевидного плана) список задач; одна **логическая задача** = один шаг (например: «Prisma-схема + первая миграция», «POST /auth/register и login», «экран логина на фронте» — отдельные шаги, а не всё сразу).
2. **Один шаг за раз** — в рамках текущего ответа реализовать **только одну** согласованную задачу от начала до проверяемого результата (код, команды, краткая сводка изменений).
3. **Стоп и вопрос** — после завершения шага: кратко описать, **что сделано** и **как проверить**; затем **явно спросить пользователя**: продолжать ли со следующей задачей.
4. **Без самовольного продолжения** — следующую задачу **не начинать**, пока пользователь не ответит утвердительно (или не даст явную команду «сделай следующие N шагов» — тогда можно выполнить ровно N шагов с остановкой после N-го).

Исключение: пользователь в одном сообщении чётко просит «сделай шаги A, B, C подряд» — выполнить не больше запрошенного, затем снова спросить про продолжение.
